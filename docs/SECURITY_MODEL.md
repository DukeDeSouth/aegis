# Модель безопасности

Дополняет [`TRUST_DOMAINS.md`](TRUST_DOMAINS.md) (что чем изолировано) описанием дефолтов и гейтов (как принимаются решения о допуске).

## Принципы дефолтов

Aegis инвертирует permissive-дефолты OpenClaw/Hermes:

- **Fail-closed.** Недоступен валидатор, broker или карантин → действие отклоняется, а не пропускается. (Инверсия `tirith_fail_open` Hermes.)
- **Deny-by-default.** Канал не отвечает никому, кроме спаренного владельца. Навык не получает capability, пока не запросил её явно в манифесте. Sandbox не имеет сети, пока маршрут не разрешён.
- **Нет глобального «yolo».** Строгость регулируется по классу действия/потока, а не одним переключателем, выключающим всё.

## Градуированная цена действия

Вместо бинарного «спрашивать / не спрашивать» — три класса:

| Класс | Примеры | Гейт |
|---|---|---|
| read-only в sandbox | прочитать файл, посчитать, распарсить | без approval |
| внешний эффект, обратимый | отправить сообщение, создать черновик | по capability-манифесту навыка |
| необратимое / дорогое / доступ к секрету | оплата, удаление, доступ к креду, отправка данных наружу | human-gate + verified-доказательство |

Это разрешает конфликт «безопасность против автономности»: дешёвые обратимые действия идут свободно, человек подключается только там, где цена ошибки высока.

## Гейт-движок

- Каждое запрошенное действие классифицируется по таблице выше.
- Решение принимается на уровне доверенного ядра, не внутри sandbox.
- Все решения пишутся в tamper-evident audit log.
- Для класса «необратимое» требуется явное подтверждение владельца и знание/навык в статусе `verified`.

## Провенанс как гейт полномочий

Полномочия действия зависят от происхождения инициирующих данных:

- инициатор — команда владельца → полный control flow;
- инициатор — недоверенный контент (карантин) → не может запускать инструменты в том же ходе;
- знание, порождённое в карантине/фоне → пониженное доверие, не самоподтверждается.

## Что чем защищено (сводка)

| Угроза | Основная защита | Вторичная |
|---|---|---|
| Prompt injection во входящем контенте | карантин (Q-LLM без инструментов) | провенанс-гейт |
| Кража секретов при компрометации | broker (агент не держит ключ) | deny-all egress в sandbox |
| Выход исполнения за пределы | контейнер (граница ОС) | non-root, allowlist mount |
| Отравление памяти | эпистемические статусы + promotion-гейт | provenance, snapshot/rollback |
| Вредоносный навык | навыки как данные; код в sandbox | capability-манифест, git-pinned, скан agent-created |
| Перерасход/DoS по токенам | budget engine + деградация | приоритет интерактива над фоном |

## Границы модели (честно)

- Изоляция полагается на корректность контейнер-рантайма и ядра ОС — общий kernel остаётся общей поверхностью атаки (как отмечают и авторы credential-brokering).
- Карантин не защищает от инъекций, нацеленных на сам вывод Q-LLM, если оркестратор действует по недоверенным данным напрямую — поэтому правило «данные не инициируют tool-call» критично и должно тестироваться.
- Определение «владелец» = идентичность канала; компрометация аккаунта владельца — отдельный класс угроз (см. THREAT_MODEL, требует 2FA на необратимое).
