# 0001. Четыре trust-домена вместо монолита

- **Статус:** Accepted
- **Дата:** 2026-07-05

## Контекст

Агенты класса (OpenClaw, Hermes) держат в одном процессе недоверенный вход, секреты и исполнение — «lethal trifecta». Prompt injection в такой конфигурации ведёт к эксфильтрации секретов и выполнению кода. Защита у них — проверки внутри процесса (регэкспы, approval), а единственная реальная граница (изоляция ОС) опциональна. NanoClaw вынес исполнение и секреты, но не вход.

## Решение

Разнести три фактора trifecta по отдельным доменам с границами на уровне ОС:

1. **Quarantine** — недоверенный вход (Q-LLM без инструментов).
2. **Sandbox** — исполнение (контейнер, deny-all egress).
3. **Broker** — секреты (отдельный процесс, proxy-инжекция).
4. **Host** — доверенное маленькое ядро (оркестратор, память, гейты).

## Последствия

**Плюсы:** injection не даёт RCE и кражи секретов структурно, а не эвристически; разделены все три фактора (Unit 42 требует хотя бы один).

**Минусы:** сложность и задержка (несколько процессов, hops). Митигация — свой код только для Host; sandbox/broker/quarantine из готовых компонентов; крайняя деградация до 3 доменов с пометкой снижения гарантий (см. 0004).

## Альтернативы

- Монолит с проверками — отвергнут (это и есть дефект класса).
- Только контейнеризация исполнения (как NanoClaw) — недостаточно: вход не изолирован.
