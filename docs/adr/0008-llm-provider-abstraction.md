# 0008. LLM: OpenAI-совместимый протокол, тонкий собственный клиент

- **Статус:** Accepted
- **Дата:** 2026-07-05

## Контекст

ARCHITECTURE.md требует model-agnostic подключение LLM через OpenAI-совместимый API, с возможно разными моделями для P-LLM и Q-LLM. Budget engine (TOKEN_ECONOMY) требует точного учёта токенов каждого вызова. Экосистема 2026 предлагает три пути: тяжёлые SDK-фреймворки (Vercel AI SDK и т.п.), gateway-прокси (LiteLLM, OpenRouter) и лёгкие библиотеки-трансляторы. Ядро ограничено ~4K LOC и принципом аудируемости (ADR-0006).

## Решение

**Протокол — OpenAI Chat Completions** как lingua franca: поддерживается всеми крупными провайдерами и локальными рантаймами (Ollama, LM Studio); переключение провайдера = смена `base_url` + `model` в конфиге.

**Реализация — собственный тонкий клиент** (~150–300 LOC, поверх нативного `fetch`, без provider-SDK в ядре). Контракт:

- `complete(request) → { message, toolCalls?, usage }`;
- `usage = { promptTokens, completionTokens, estimated: boolean }` — **обязательное** поле; если провайдер не вернул usage, клиент подставляет верхнюю оценку (промпт — по оценке символов, completion — `max_tokens`) с `estimated: true`. Budget engine различает точный учёт и оценку; вызов никогда не проходит «мимо бюджета» (fail-closed);
- retry/backoff на транзиентных ошибках; идемпотентность обеспечивается на уровне очереди, не клиента;
- стриминг не входит в MVP (ответы в мессенджер уходят целыми сообщениями).

**Конфигурация — два профиля:**

```
llm:
  p_llm: { base_url, model, key_ref, max_tokens }   # оркестратор: tools разрешены
  q_llm: { base_url, model, key_ref, max_tokens }   # карантин: запрос структурно без tools
```

- Тип запроса Q-LLM не содержит поля `tools` (структурная гарантия ADR-0005, а не проверка в рантайме).
- `key_ref` — ссылка на секрет, не значение: ключ инжектируется окружением/broker (ADR-0004), в конфиге и логах не появляется.

**Провайдер по умолчанию не фиксируется.** Документация развёртывания приводит проверенные конфигурации: OpenRouter (один ключ — много моделей) и Ollama (локально, без внешнего трафика). P-LLM и Q-LLM могут указывать на разных провайдеров с первого дня — это свойство конфигурации.

## Последствия

**Плюсы:** ноль тяжёлых зависимостей в доверенном ядре; полный контроль над usage-учётом; смена провайдера без кода; Q-LLM-ограничение выражено типом.

**Минусы и митигации:**

- Свой клиент = свой код на поддержку (~300 LOC) — приемлемо против неаудируемых транзитивных деревьев SDK.
- Провайдер-специфичные расширения (Anthropic thinking, structured outputs) недоступны через общий протокол — осознанно: MVP они не требуются; при необходимости — точечное расширение клиента новым ADR.
- Некоторые OpenAI-совместимые endpoint'ы возвращают неточный/пустой usage — закрыто правилом верхней оценки (`estimated: true`).

## Альтернативы

- **Vercel AI SDK / LangChain** — отвергнуто: большие транзитивные деревья в доверенном ядре, LOC/аудит-бюджет не проходят.
- **LiteLLM-прокси как обязательный компонент** — отвергнут: лишний процесс и хоп для single-tenant агента; не даёт ничего, чего нет у тонкого клиента.
- **Официальный `openai` SDK с baseURL** — рабочий, но добавляет зависимость ради подмножества функций; собственный клиент того же размера прозрачнее. Допустим как отправная точка реализации, если сэкономит время Sprint 1.
