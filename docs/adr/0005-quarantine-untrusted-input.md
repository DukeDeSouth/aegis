# 0005. Карантин недоверенного входа

- **Статус:** Accepted
- **Дата:** 2026-07-05

## Контекст

Даже с изоляцией исполнения и секретов недоверенный контент (письма, веб, файлы) в продуктах класса течёт в тот же LLM-контекст, что и команды владельца. Это оставляет prompt injection работоспособным. CaMeL (Google DeepMind) показал структурное решение — разделение control/data flow, — но существует только как research-артефакт.

## Решение

Недоверенный контент (не от владельца) обрабатывает отдельный Q-LLM без доступа к инструментам. Правило: **недоверенные данные могут войти в рассуждение оркестратора, но не могут инициировать вызов инструмента в том же ходе.**

Прямые команды владельца карантин минует (доверенный control flow), чтобы не платить задержкой и потерей смысла на каждом сообщении.

## Последствия

**Плюсы:** injection из контента не приводит к действию структурно; позиция в классе свободна (никто не карантинит вход).

**Минусы:**

- Задержка dual-LLM — митигируется тем, что карантин только для недоверенного контента.
- «От владельца» = идентичность канала → захват аккаунта обходит карантин. Митигация — provenance + 2FA на необратимое (пост-MVP).
- Соблазн упростить до «извлекать только структуру» сломал бы free-text-задачи (ресёрч, почта) — поэтому формулировка именно «данные без права инициировать действие», а не «только структура».

## Альтернативы

- Весь вход в один контекст (класс) — отвергнуто (injection работает).
- Полный CaMeL-интерпретатор на весь поток — избыточно для personal-агента (−7 п.п. utility в бенчмарке), применяем прагматичную версию только к недоверенному входу.
