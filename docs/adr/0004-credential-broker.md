# 0004. Секреты через broker, не в процессе агента

- **Статус:** Accepted
- **Дата:** 2026-07-05

## Контекст

Если агент держит секрет (env, файл, сокет), тот же процесс может секрет и использовать, и раскрыть — а значит, prompt injection или скомпрометированный навык уводят ключ. Индустрия 2026 сошлась на паттерне credential brokering (IETF draft CB4A, CapSeal, Envoy-sidecar, реализации Anthropic/Vercel/Cloudflare/LangChain).

## Решение

Агент никогда не получает реальный секрет. Отдельный процесс-broker:

- перехватывает исходящие запросы и подставляет учётные данные на уровне прокси;
- выдаёт короткоживущие scoped-хендлы под конкретную операцию;
- применяет политику и ведёт tamper-evident аудит.

Sandbox имеет deny-all egress; весь трафик идёт через broker-proxy.

## Последствия

**Плюсы:** секрет, который агент не видел, нельзя выманить инъекцией; аудит и scope из коробки.

**Минусы:** broker и агент на общем kernel — уязвимость ядра снимает границу (отмечено и авторами паттерна). Митигация — вынос broker на отдельный хост после MVP.

**Деградация для MVP:** broker допустимо запустить как локальный proxy; крайний случай для соло-разработчика — временно совместить с Host с явной пометкой снижения гарантий, с планом вынести.

## Альтернативы

- Секреты в env/файле (OpenClaw/Hermes дефолт) — отвергнуто.
- Vault без proxy-инжекции (агент всё же получает токен) — слабее; broker предпочтительнее.
