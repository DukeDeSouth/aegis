# Верифицируемое обучение

Главный дифференциатор Aegis. Агент накапливает знания и навыки (как Hermes), но, в отличие от него, ничего выученного не влияет на будущие сессии, пока это не проверено.

## Два уровня памяти

### Эпизодическая
Все сессии сохраняются и доступны полнотекстовым поиском **без обращения к LLM** (дёшево, всегда доступно). Служит для «вспомнить, что было» и для сбора доказательств. Аналог — FTS-поиск по журналу сессий.

### Семантическая (знания и навыки)
Каждая запись несёт метаданные:

- `epistemic_status` ∈ `{unverified, corroborated, verified, refuted}`
- `provenance` — кто и в каком домене породил: `owner` / `orchestrator` / `quarantine` / `background`
- `evidence[]` — ссылки на то, что подтвердило запись (успешный тест, воспроизведённое наблюдение, подтверждение владельца)

## Жизненный цикл знания

```
агент предложил знание/навык
   → status = unverified, лежит в staging, НЕ в системном контексте
        │
        ├─ прошла детерминированная проверка (тест / dry-run / повтор наблюдения)  → corroborated
        ├─ подтверждено независимым источником или владельцем                      → verified
        └─ обнаружено противоречие                                                → refuted (не грузится)

promotion-гейт:
   corroborated → попадает в системный контекст с пометкой доверия (автономно, без владельца)
   verified     → требуется для действий класса «необратимое / дорогое»
```

## Почему это закрывает два минуса класса

- **Self-poisoning.** Отравленная или обманутая сессия может записать знание, но оно рождается `unverified` с provenance `quarantine`/`background` и без прохождения гейта не попадёт в системный контекст. Одна скомпрометированная сессия не отравляет будущее.
- **Накопление без верификации.** Ошибочное знание не «компаундится»: пока оно не проверено, оно не влияет ни на что.

## Баланс «безопасно, но не тупо»

Строгий гейт рискует зеркальным провалом Hermes: если в контекст попадает только то, что подтвердил человек, агент будет казаться «тупым». Разрешается **градацией**:

- Воспроизводимые знания (например, «команда Y успешно собрала проект» — проверяемо повтором/тестом) становятся `corroborated` **автоматически, без владельца**, и сразу доступны.
- Человек нужен только для `verified`, а `verified` требуется только для необратимых/дорогих действий.

Так автономность сохраняется для дешёвых знаний, а жёсткий гейт стоит только там, где цена ошибки высока.

## Курация

Двухфазная, с инверсией дефолтов Hermes:

- **Детерминированная фаза (всегда, бесплатно):** staleness (устаревание), usage-счётчики, дедупликация, decay. Работает без LLM.
- **LLM-консолидация (только по бюджету):** обобщение/переписывание — опционально, списывается с дневного бюджета.

Перед любой мутацией хранилища — **snapshot/rollback** (наследуется у Hermes Curator), чтобы плохую консолидацию можно было откатить.

Рефлексия («не пора ли что-то сохранить/пересмотреть») запускается **по событию** (коррекция владельца, повтор ошибки), а не по счётчику ходов «обязательно сохрани что-нибудь» — последнее у Hermes генерирует мусор, под который потом нужна отдельная чистка.

## Метрика

`reuse_rate` — доля сохранённых знаний, реально переиспользованных в последующих сессиях. Знание, которое не переиспользуется, — кандидат на decay. Self-improvement, не поднимающий reuse_rate, отключается (см. TOKEN_ECONOMY).

## Открытые вопросы для реализации

- Точные критерии «детерминированной проверки» для разных типов знаний.
- Схема хранения evidence и защита её от раздувания и отравления.
- Как показывать владельцу staging (что ждёт подтверждения) без спама.
