# Спринт-план

Разбивка фаз из [`../ROADMAP.md`](../ROADMAP.md) на спринты по 2 недели. Ориентир для малой команды (1–3 человека), не обязательство по датам. Каждый спринт имеет одну цель, задачи и Definition of Done. Спринты идут последовательно — принцип «сначала границы, потом удобства» сохранён.

Оценки в скобках — грубые (S ≈ 1–2 дня, M ≈ 3–4 дня, L ≈ неделя+).

---

## Sprint 0 — Фундамент решений (Фаза 0)

**Цель:** закрыть проектные развилки, блокирующие код.

- [ ] ADR-0006: язык ядра и рантайм sandbox (M)
- [ ] ADR-0007: формат capability-манифеста навыка (S)
- [ ] Схема данных памяти: таблицы эпизодической + семантической (статусы, provenance, evidence) (M)
- [ ] Скелет репозитория: структура каталогов, линтер, CI-заготовка, тест-раннер (S)
- [ ] Выбор LLM-провайдера и абстракция вызова (OpenAI-совместимый) (S)

**DoD:** нет открытых вопросов, блокирующих Sprint 1; схема памяти зафиксирована в миграции; `hello world`-пайплайн CI зелёный.

---

## Sprint 1 — Ядро и очереди (Фаза 1, часть 1)

**Цель:** сообщение проходит через ядро от входа к выходу.

- [ ] Host: inbound/outbound очереди на SQLite (M)
- [ ] Петля оркестратора: забор из очереди → вызов LLM → ответ в очередь (L)
- [ ] Audit log (append-only, tamper-evident) (S)
- [ ] Конфиг и запуск процесса (S)

**DoD:** локально «эхо-агент» принимает сообщение из очереди, зовёт LLM, пишет ответ; каждое действие в audit log.

---

## Sprint 2 — Канал и авторизация (Фаза 1, часть 2)

**Цель:** агент доступен из Telegram только владельцу.

- [ ] Channel Adapter для Telegram (официальный Bot API) (M)
- [ ] Deny-by-default авторизация + pairing владельца (M)
- [ ] Провенанс на входе: пометка «от владельца» vs «недоверенное» (S)
- [ ] Обработка ошибок канала, retry/backoff (S)

**DoD:** владелец пишет боту в Telegram и получает ответ; чужие сообщения отклоняются; провенанс проставляется на каждом входящем.

---

## Sprint 3 — Sandbox и Broker (Фаза 1, часть 3)

**Цель:** исполнение и секреты вынесены за границу ОС.

- [ ] Sandbox исполнения (Docker): deny-all egress, allowlist mount, non-root (L)
- [ ] Credential Broker (локальный proxy): proxy-инжекция, агент не видит ключ (L)
- [ ] Трафик sandbox только через broker-proxy (M)

**DoD:** тест V2 (агент не достаёт сырой секрет ни в одной точке) и V3 (код в sandbox не выходит в сеть мимо broker и не читает хост) — проходят.

---

## Sprint 4 — Гейт действий (Фаза 1, финал)

**Цель:** каждое действие проходит через градуированный гейт, fail-closed.

- [ ] Gate engine: классы read-only / обратимое / необратимое (M)
- [ ] Провенанс как гейт полномочий (недоверенные данные не инициируют tool-call) (M)
- [ ] Fail-closed поведение при недоступности проверки/broker (S)
- [ ] Human-gate для необратимых действий (подтверждение в чате) (S)

**DoD:** read-only идёт свободно; необратимое требует подтверждения; при «упавшем» валидаторе действие отклоняется. **Веха: скелет с границами готов.**

---

## Sprint 5 — Память (Фаза 2, часть 1)

**Цель:** агент помнит и ищет без LLM.

- [ ] Эпизодическая память: запись сессий + полнотекстовый поиск (M)
- [ ] Семантическая память: запись знаний со статусами и provenance (M)
- [ ] Инжекция verified/corroborated знаний в системный контекст (M)

**DoD:** агент находит прошлые сессии поиском без вызова LLM; знания хранятся со статусом и источником.

---

## Sprint 6 — Обучение и курация (Фаза 2, финал)

**Цель:** знание проходит верификацию, память не отравляется и не пухнет.

- [ ] Promotion-гейт: corroborated автоматически, verified для необратимого (M)
- [ ] Детерминированная проверка знаний (тест/повтор наблюдения → corroborated) (M)
- [ ] Курация: staleness, usage, дедуп, decay (M)
- [ ] Snapshot/rollback перед мутациями хранилища (S)

**DoD:** тест V4 (знание из недоверенного источника не попадает в контекст без промоушена) проходит; воспроизводимое знание становится corroborated без владельца.

---

## Sprint 7 — Карантин входа (Фаза 3)

**Цель:** недоверенный контент не может инициировать действие.

- [ ] Quarantine-плоскость: Q-LLM без доступа к инструментам (L)
- [ ] Правило «недоверенные данные входят в рассуждение, но не запускают tool-call в том же ходе» (M)
- [ ] Обработка пересланного контента, веб-страниц, вложений (M)

**DoD:** тест V1 (prompt injection из пересланного письма не приводит к вызову инструмента) проходит; при этом free-text задача «прочитай и порассуждай» работает.

---

## Sprint 8 — Навыки (Фаза 4, часть 1)

**Цель:** агент расширяется навыками-данными безопасно.

- [ ] Декларативные навыки: markdown + capability-манифест (M)
- [ ] Прогрессивное раскрытие (list → view) (S)
- [ ] Навыки с кодом через sandbox + verifiable loop (dry-run → corroborated) (L)
- [ ] Установка из git с pinned-версией; скан agent-created кода (M)

**DoD:** декларативный навык применяется; код-навык проходит dry-run в sandbox перед допуском; `curl|bash` и runtime-install в хосте невозможны.

---

## Sprint 9 — Автоматизации и бюджет (Фаза 4 финал + Фаза 5, часть 1)

**Цель:** расписания работают, токены под контролем.

- [ ] Scheduler: cron-задачи как сообщения во входную очередь (M)
- [ ] Budget engine: дневной лимит, приоритет интерактива над фоном (M)
- [ ] Явная деградация при исчерпании + уведомление владельцу (S)

**DoD:** cron-задача наследует те же гейты, что обычный ввод; тест V7 (исчерпание бюджета даёт уведомление + деградацию, не тихий сбой) проходит.

---

## Sprint 10 — Метрики, закалка, релиз MVP (Фаза 5 финал)

**Цель:** MVP пригоден к самостоятельному развёртыванию.

- [ ] Метрика reuse_rate + отключение бесполезного self-improvement (M)
- [ ] Прогон всех тестов из [`MVP_SCOPE.md`](MVP_SCOPE.md) (критерии готовности) (M)
- [ ] Проверка размера ядра против цели ~4K LOC (S)
- [ ] Документация по развёртыванию self-hosted (M)

**DoD:** все критерии готовности MVP из `MVP_SCOPE.md` отмечены; владелец может развернуть агента по документации. **Веха: MVP.**

---

## После MVP (бэклог, без спринтов)

- Дополнительные каналы (по одному, только официальные API)
