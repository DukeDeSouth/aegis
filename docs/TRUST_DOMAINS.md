# Четыре trust-домена

Основа безопасности Aegis. Каждый из трёх факторов lethal trifecta (недоверенный вход, секреты, исполнение) вынесен в отдельный домен с границей на уровне ОС. Unit 42 требует структурно разделить хотя бы один фактор — Aegis разделяет все три.

## Домен 1 — Quarantine (недоверенный вход)

**Что изолирует:** контент, пришедший не от владельца — пересланные письма, веб-страницы, вложения, вывод инструментов.

**Как:** отдельный LLM (Q-LLM) без доступа к инструментам обрабатывает этот контент. Результат возвращается оркестратору помеченным как недоверенные данные. Публичный аналог — CaMeL (разделение control/data flow, Google DeepMind, arXiv 2503.18813).

**Ключевое правило:** недоверенные данные _могут войти в рассуждение_ оркестратора, но _не могут стать источником вызова инструмента в том же ходе_. Это не «извлечь только структуру» — иначе ломаются задачи вроде «прочитай статью и порассуждай». Это «данные без права инициировать действие».

**Что НЕ карантинится:** прямые команды владельца — это доверенный control flow, он идёт прямо в оркестратор. Иначе карантин добавил бы задержку и потерю смысла на каждом сообщении.

**Граница-риск:** определение «от владельца» привязано к идентичности в канале → захват аккаунта обходит карантин. Митигация — provenance по источнику + 2FA на необратимые действия (см. THREAT_MODEL).

## Домен 2 — Sandbox (исполнение)

**Что изолирует:** любой shell, доступ к файлам, исполнение кода навыков.

**Как:** контейнер (Docker / Apple Container / micro-VM) с:

- политикой **deny-all** на исходящую сеть — весь трафик только через broker-proxy;
- явно смонтированными директориями (allowlist) — ничего сверх разрешённого;
- запуском от non-root;
- недоступностью состояния хоста (память, очереди, аудит).

**Почему контейнер, а не проверки:** даже если агент/навык внутри получит root, он не выйдет за границу контейнера — её держит ядро ОС, а не `if` в коде. Публичный аналог — модель NanoClaw.

## Домен 3 — Credential Broker (секреты)

**Что изолирует:** API-ключи, токены, SSH-креды.

**Как:** отдельный процесс (в MVP — локальный proxy). Агент никогда не получает реальный ключ; broker:

- перехватывает исходящие запросы и подставляет учётные данные на уровне прокси;
- выдаёт короткоживущие scoped-хендлы под конкретную операцию;
- применяет политику (какому запросу что можно) и ведёт tamper-evident аудит.

Публичные аналоги — credential brokering (IETF draft CB4A), CapSeal (arXiv 2604.16762), Envoy-sidecar-инжекция.

**Почему это работает:** секрет, который агент никогда не видел, невозможно выманить через prompt injection.

## Домен 4 — Host (доверенное ядро)

**Что содержит:** router, входную/выходную очереди, память, планировщик, gate engine, budget engine, audit log, оркестратор (P-LLM).

**Свойство:** это единственный компонент, которому мы доверяем, поэтому он намеренно мал (цель ≤ ~4K LOC) и минимален по зависимостям — его реально можно прочитать и проверить. Навыки и адаптеры каналов живут вне ядра.

## Матрица доверия

| Домен      | Доверяет                             | Не доверяет                                                |
| ---------- | ------------------------------------ | ---------------------------------------------------------- |
| Host       | командам владельца, verified-знаниям | недоверенному контенту, unverified-знаниям, выводу sandbox |
| Quarantine | ничему                               | всему входящему контенту                                   |
| Sandbox    | смонтированным данным                | сети (deny-all), собственному коду навыков                 |
| Broker     | политике и идентичности запроса      | агенту (он клиент, не владелец кредов)                     |

## Реализуемость

Свой код — только Host. Sandbox — готовый контейнер-рантайм; Broker — готовый proxy; Quarantine — один дополнительный LLM-вызов без собственного состояния. Аудируемость относится к доверенному ядру, а не к сумме всего стека. Крайняя деградация для соло-разработчика: объединить Broker в Host на старте с явной пометкой снижения гарантий (см. ADR-0004).
