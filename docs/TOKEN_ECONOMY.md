# Экономика токенов

Продукты класса склонны к обратной экономике: фоновые self-review, ансамбли, рефлексия по счётчику — жгут токены с трудноизмеримым выхлопом. Aegis инвертирует это: **детерминированное бесплатно и всегда, работа LLM — по явному бюджету с отчётом о ценности.**

## Принципы

1. **Детерминированное — даром.** Полнотекстовый поиск по памяти, счётчики, staleness, дедупликация, курация по правилам — без LLM, всегда доступны, в бюджет не входят.
2. **LLM — по бюджету.** Каждый фоновый LLM-прогон (рефлексия, консолидация памяти, суммаризация) списывается с дневного лимита, задаваемого владельцем (в токенах или деньгах).
3. **Приоритет интерактива.** Интерактивные запросы владельца имеют приоритет над фоновыми задачами. Под нож при нехватке бюджета первым идёт фон.
4. **Явная деградация, не тихий сбой.** Бюджет исчерпан → агент переходит в детерминированный режим и **уведомляет** владельца, а не пропускает задачу молча и не превышает лимит незаметно.

## Сценарий деградации (пример)

Задача «каждое утро собрать дайджест почты», бюджет исчерпан к ночи:

```
03:12 — фоновая рефлексия исчерпала дневной LLM-бюджет
07:00 — cron «дайджест»:
        LLM-бюджет = 0
        → агент шлёт владельцу:
          «Дайджест не собран: дневной LLM-бюджет исчерпан в 03:12.
           Поднять лимит / перенести / пропустить?»
        → как fallback прикладывает выборку писем полнотекстовым поиском (без LLM)
```

Владелец видит проблему и причину, а не молчание.

## Метрика ценности

`reuse_rate` — доля сохранённых знаний, реально переиспользованных в последующих сессиях. Идея: self-improvement оправдан только если он повышает переиспользование.

- Знание, которое не переиспользуется, — кандидат на decay.
- Вид фонового self-improvement, не поднимающий reuse_rate, **отключается**.

## Чего избегаем (антипаттерны Hermes)

- Рефлексия по счётчику ходов «обязательно сохрани что-нибудь каждые N сообщений» — генерирует мусор.
- Обязательные фоновые ансамбли/само-ревью на каждый ход.
- Тихий перерасход без учёта и отчёта.

## Открытые вопросы

- Единица бюджета: токены, деньги или оба; как оценивать до вызова.
- Разделение бюджетов: интерактив / фон / консолидация — отдельные лимиты или один общий с приоритетами.
- Как считать reuse_rate дёшево и честно.
