# Контрибуции в Aegis

Спасибо за интерес к проекту. Aegis — security-first агент, поэтому у контрибуций есть несколько жёстких правил. Они короткие, но обязательные.

## Быстрый старт

```bash
# Node LTS — версия закреплена в .nvmrc
npm ci --ignore-scripts && npm rebuild better-sqlite3
npm run lint && npm run typecheck && npm test
```

Опционально (нужен запущенный Docker):

```bash
npm run test:security   # V2/V3: границы sandbox/broker на реальном Docker
```

Прогон перед PR — тот же набор, что и в CI:

```bash
npm run lint && npm run format:check && npm run typecheck && npm test && npm run loc
```

## Правила проекта

### 1. Ядро остаётся маленьким

`npm run loc` считает непустые строки TypeScript в `src/`. Порог — **4000 строк**. Это не формальность, а MVP-критерий: ядро должно читаться и аудироваться за разумное время. Если фича не влезает — сначала обсуждение в issue, возможно, ей место в навыке (`skills/`), а не в ядре.

### 2. Закрытый список зависимостей

Runtime-зависимости `src/` — закрытый список (`better-sqlite3`, `zod`; см. [`docs/REPO_LAYOUT.md`](docs/REPO_LAYOUT.md)). Добавление новой зависимости — это архитектурное решение: открывайте issue до написания кода. Install-скрипты запрещены (`--ignore-scripts`), единственное исключение — `npm rebuild better-sqlite3`.

### 3. Security-инварианты не обсуждаются в PR

Границы trust-доменов, fail-closed поведение гейта, карантин недоверенного входа, эпистемические статусы памяти — описаны в [`docs/SECURITY_MODEL.md`](docs/SECURITY_MODEL.md) и [`docs/TRUST_DOMAINS.md`](docs/TRUST_DOMAINS.md). PR, ослабляющий инвариант «для удобства», будет отклонён. Если считаете инвариант ошибочным — сначала issue с обоснованием, изменение оформляется как ADR.

### 4. Архитектурные решения — через ADR

Существенные изменения архитектуры (новый trust-домен, замена рантайма, формат манифестов и т.п.) фиксируются в [`docs/adr/`](docs/adr/) по образцу существующих записей.

### 5. Тесты обязательны

- Новая логика — unit-тесты (`test/unit/`).
- Поведение через очереди/оркестратор — интеграционные тесты (`test/integration/`).
- Изменения, касающиеся границ безопасности, — тесты в `test/security/` (гоняются на реальном Docker).

## Процесс

1. **Issue сначала** — для всего, что больше тривиального фикса. Это экономит время всем: возможно, фича противоречит модели угроз или уже запланирована иначе.
2. **Форк → ветка → PR.** Один PR — одно логическое изменение.
3. **CI должен быть зелёным**: lint, format, typecheck, tests, security-tests, audit, loc.
4. **Описание PR** — что и почему; для изменений поведения — как проверить руками.

## Стиль кода

- TypeScript strict (`noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`) — без `any` и подавлений без причины.
- Форматирование — Prettier (`npm run format`), линт — ESLint type-checked.
- Комментарии объясняют «почему», а не «что».

## Уязвимости

Не открывайте публичные issue для уязвимостей — см. [`SECURITY.md`](SECURITY.md).
