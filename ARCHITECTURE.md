# Архитектура Aegis

> Живой документ. Отражает целевую архитектуру, а не текущий код (кода пока нет).

## Несущая идея

Безопасность Aegis — это топология, а не набор проверок. Три вещи, которые в других агентах живут в одном процессе (недоверенный вход, секреты, исполнение), здесь разнесены по отдельным доменам с границами на уровне операционной системы. Компонент, которому мы доверяем, — только маленькое ядро; всё остальное считается потенциально враждебным и изолируется.

## Схема

```
                    Telegram (официальный Bot API)
                              │
   ┌────────────────── HOST — доверенное ядро (цель ≤ ~4K LOC) ──────────────────┐
   │                                                                              │
   │   Channel Adapter ─▶ inbound queue (SQLite) ─▶ Orchestrator (P-LLM)          │
   │                                                     │                        │
   │        Memory store   Scheduler   Gate engine   Budget engine   Audit log    │
   │                                                     │                        │
   └───┬──────────────────────────┬──────────────────────┬────────────────────────┘
       │ untrusted-контент          │ control flow          │ нужен секрет / сеть
       ▼                            ▼                       ▼
┌──────────────────┐   ┌──────────────────────────┐   ┌────────────────────────────┐
│ QUARANTINE        │  │ SANDBOX                   │   │ CREDENTIAL BROKER           │
│ Q-LLM, без tools  │  │ контейнер, no secrets,    │   │ отдельный процесс:          │
│ untrusted-данные  │─▶│ deny-all egress,          │──▶│ инжектит креды на proxy,    │
│ не инициируют     │  │ только смонтированные dirs│   │ scoped + TTL + policy,      │
│ tool-call         │  │ трафик — через broker     │   │ audit; агент не видит ключ  │
└──────────────────┘   └──────────────────────────┘   └────────────────────────────┘
```

## Компоненты

### Host — доверенное ядро
Единственный компонент, которому мы доверяем, и потому он намеренно маленький и читаемый. Держит:
- **Channel Adapter** — тонкий приёмник сообщений из мессенджера. Не содержит бизнес-логики агента.
- **Inbound/Outbound queue** — файловые очереди (SQLite). Все входящие события, включая задачи планировщика, проходят через одну входную очередь — привилегированного «внутреннего» пути нет.
- **Orchestrator (P-LLM)** — привилегированный LLM. Видит команды владельца (доверенный control flow) и результаты, но не сырой недоверенный контент.
- **Memory store** — эпизодическая (полнотекстовый поиск) и семантическая (знания/навыки со статусами) память. См. `docs/LEARNING_LOOP.md`.
- **Scheduler** — cron; задачи публикуются как сообщения во входную очередь.
- **Gate engine** — принимает решение о допуске действия по его классу (см. `docs/SECURITY_MODEL.md`). Fail-closed.
- **Budget engine** — учёт токенов, деградация при исчерпании. См. `docs/TOKEN_ECONOMY.md`.
- **Audit log** — tamper-evident журнал решений и действий.

### Quarantine — карантин недоверенного входа
Отдельный LLM (Q-LLM) без доступа к инструментам обрабатывает контент, пришедший не от владельца: пересланные письма, веб-страницы, файлы, вывод инструментов. Публичный аналог подхода — CaMeL (разделение control/data flow, Google DeepMind).

Ключевой принцип: **недоверенные данные могут войти в рассуждение оркестратора, но не могут стать источником вызова инструмента в том же ходе.** Это не «извлечь только структуру» (иначе ломается ресёрч и работа с почтой), а «данные без права инициировать действие».

### Sandbox — изоляция исполнения
Любой shell/файловый доступ/исполнение кода — только внутри контейнера с политикой deny-all на исходящую сеть и явно смонтированными директориями. Граница — ядро ОС (контейнер/micro-VM), а не проверки в коде. Состояние хоста (память, очереди) из sandbox напрямую недоступно. Весь сетевой трафик sandbox идёт через broker-proxy.

### Credential Broker — изоляция секретов
Агент никогда не держит реальный ключ. Broker в отдельном процессе перехватывает исходящие запросы, подставляет учётные данные на уровне прокси, выдаёт короткоживущие scoped-хендлы и ведёт аудит. Публичные аналоги — credential brokering (IETF draft CB4A, CapSeal, Envoy-sidecar).

## Потоки данных (кратко)

1. **Команда владельца → действие.** Сообщение → inbound queue → Orchestrator строит план → действие проходит Gate engine по классу → при необходимости секрета/сети идёт через Broker → результат → outbound queue → канал.
2. **Недоверенный контент.** Пересланное/веб/файл → Quarantine (Q-LLM) → типизированные/помеченные данные → Orchestrator использует их в рассуждении, но инструменты в этом ходе не инициируются недоверенными данными.
3. **Обучение.** Агент предлагает знание/навык → статус `unverified`, staging → детерминированная проверка → `corroborated`/`verified` → promotion-гейт → попадает в системный контекст будущих сессий. См. `docs/LEARNING_LOOP.md`.
4. **Расписание.** Scheduler публикует задачу как сообщение во входную очередь — она наследует те же гейты, что и обычный ввод.

## Границы (что чему доверяет)

| Компонент | Доверяет | Не доверяет |
|---|---|---|
| Host/Orchestrator | командам владельца, verified-знаниям | недоверенному контенту, unverified-знаниям, выводу sandbox |
| Quarantine | ничему (обрабатывает враждебный вход) | всему входящему контенту |
| Sandbox | смонтированным данным | сети (deny-all), собственному коду навыков |
| Broker | политике и идентичности запроса | агенту (агент — клиент, не владелец кредов) |

## Технологические заметки (не финал)

- Язык ядра — TBD (кандидаты: TypeScript для скорости разработки; Rust/Go для broker). Решение — отдельным ADR.
- Sandbox — готовые кирпичи (Docker / Apple Container / micro-VM), не собственная реализация.
- Broker в MVP может быть локальным proxy; вынос на отдельный хост — позже.
- LLM — model-agnostic через OpenAI-совместимый API; Q-LLM и P-LLM могут быть разными моделями.

Принцип реализуемости малой командой: **свой код — только доверенное ядро.** Broker, sandbox, карантин собираются из готовых компонентов. «Аудируемость» относится к ядру, а не к сумме всего стека.
